# Agent learnings - 2025-08-09

## TypeScript and code quality preferences

### Type safety

- **Never use type assertions** - JSON.parse is always unsafe and should be validated with proper type guards
- **Never use the `any` type** - Always properly type or use `unknown` with validation
- **Prefer `undefined` over `null`** unless there's a compelling reason (like serialization needs)

### Code structure

- **Function complexity should be reduced by extracting single-responsibility functions**
- **Use functional patterns** like map/filter over imperative loops when appropriate
- **Prefer `undefined` returns over `null`** for optional values
- **Keep types co-located with functions** - Only create separate types.ts if there's no clear tree-like architecture. Types should live with the functions that create or primarily use them
- **Don't write complicated functions that can be solved by lightweight libraries** - Use established libraries like `semver` instead of custom parsing. Even if it's a transitive dependency, add it as a direct dependency for clarity

### Command execution patterns

- **Always use `pnpm run {script}`** for package.json scripts
- **Always use `pnpm exec {binary}`** for binary executables
- This maintains clear distinction between scripts and binaries, avoiding ambiguity

### Communication patterns

- **Pause frequently for user input** - Don't get into refactoring rabbit holes without checking in
- **Ask for guidance** on naming and approach before implementing
- **Always run linting and typechecking before considering code complete** - This is mandatory, not optional
